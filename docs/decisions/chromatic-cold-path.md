# Решение: холодный хроматический путь — serve-then-verify, бит-идентичный через fallback

Статус: **ПЕРЕСМОТРЕНО (breakpoint-таблицы НЕ строим)** · Дата: 2026-06-14
Область применения: `chromafast.rs`, `solve.rs`, `semantic.rs`.
Связано с: [`bg-dependency-invariant.md`](./bg-dependency-invariant.md) (фундамент — фон входит только через `(Y_hk, Y_wcag, J'_bg)`).

> ## ⚠️ ПЕРЕСМОТР — breakpoint-таблицы не нужны
>
> Этот документ записал serve-then-verify breakpoint-**поверхность** как путь к
> быстрому холодному хроматическому фону. **Решено иначе** — таблицы НЕ строим.
> Две находки (обе подтверждены измерениями):
>
> 1. **Формула уже оптимизирована кэшами.** Замена бисекции на безопасную секущую
>    (Ньютон) в `ratio_for_target_mp` / `match_lightness` — бит-в-бит, но реальный
>    выигрыш ~**3%**: dedup-кэш форвардов + curve-plan-кэш уже съели стоимость
>    поиска. Прототип откатан. То есть «лечить формулу» нечего, и таблицы (тысячи
>    заранее посчитанных ответов «на все случаи») — невежливая трата памяти.
>
> 2. **Настоящий рычаг для динамического фона — пересчитывать РЕЖЕ, а не быстрее**
>    (как у Apple: плавно, с задержкой; не солвер каждый кадр). Реализовано:
>    дешёвый **recheck-примитив** (`measure_contrast` / `recheck_against` /
>    WASM `recheckContrast`) + **ленивый/гистерезис-контроллер** `adaptTheme` в
>    рантайме (`packages/colors/adapt-theme.js`): каждый кадр перепроверяем, держим
>    пока проходят, пересчитываем + плавно переезжаем только при устойчивом
>    провале. Нет таблиц, нет покадрового солва, нет дрожания.
>
> **Полезное наследие (ниже остаётся в силе):** ключевое наблюдение, что per-role
> verify — это собственный acceptance-предикат солвера (1 форвард, R1 закрыт
> проверкой), и оценка стоимости verify — это ровно то, на чём стоит recheck.
> Серверный bit-identity-инвариант (golden-гейты) тоже остаётся. Сама же
> breakpoint-поверхность как способ ускорения — отложена/не нужна; держим как
> исторический разбор вариантов A/B/C.

## Проблема

`resolve_set` на **сером** фоне — O(1) (`greyfast`, ~314 ns). На **хроматическом** — полный live-солвер (~1.1 ms, ~800 CAM16-форвардов на сет). `chromafast` (#68) делает **повторный** хроматический фон O(1) мемоизацией, но **холодный** (новый фон каждый кадр — анимация/блюр) всё ещё платит полную цену. Это узкое место бесшовного динамического теминга LabUI.

## Что НЕ работает (и почему)

- **Сократить число форвардов без precompute (вариант C).** `match_lightness` уже LUT-сидит светлоту переднего плана (`solve.rs:1095`); андертон роли — малая хрома → уже идёт по быстрому `Bracket`-пути. Остаток WARM-форвардов — это per-role `finish` (`solve.rs:1217`, **по одному форварду на роль, неустранимо** — это и есть измерение, дающее итоговый hex) + curve-refine. Реалистично C: ~2–4× на COLD, **маргинально** на WARM. Не решает задачу.
- **Отдавать приближённый/ближайший кандидат (memo→near-bg, интерполяция).** Любой ответ, отличающийся от live-солвера хотя бы на 1 квант, **ломает три байт-равенственных гейта**: `resolve_set_hex_matches_golden` (вкл. хроматический `#3478F6`), `greyfast`-parity, `chromafast`-parity. Бит-идентичность здесь — **культурный инвариант**, а не случайность.

## Решение: serve-then-verify гибрид

Отдать кандидата из precompute-поверхности, затем **дёшево пере-проверить контракт каждой роли против реального фона** и при несовпадении свалиться в live-солв этой роли.

**Ключевое наблюдение (снимает риск R1 — молчаливо неверный цвет):**
per-role verify — это **собственный acceptance-предикат Stage-3 солвера** (`solve.rs:586–629`: `finish` → `meets_floor_lc` (perceptual) + `legal_ok` (WCAG-floor)). Один CAM16-форвард (`finish`, `solve.rs:1217`) + чистая WCAG-арифметика (`floor_ratio_of`, `solve.rs:970`, **0 форвардов**). Кандидат, который НЕ удовлетворяет контракт, **проваливает ровно эти проверки** → fallback. Значит:

> **Неполнота breakpoint-поверхности стоит только СКОРОСТИ (больше fallback'ов), но никогда — КОРРЕКТНОСТИ.** R1 закрыт verify, а не доказательством полноты. Бит-идентичность сохраняется тем, что fallback — это live-солвер: отданный выход байт-в-байт совпадает с live, потому что всё, что не совпало, ушло в fallback.

Это разделяет тяжёлое (доказательство полноты A) и безопасность: доказательство покупает *скорость* (меньше fallback'ов), а не *корректность* — гораздо лучшая риск-позиция.

## Стоимость verify (из кода)

| Проверка | Форварды | Источник |
|---|---|---|
| Lc/перцептивная (пере-измерить Y_hk/Lc кандидата) | **1** (`cam16::forward` в `finish`) | `solve.rs:1217–1220` |
| WCAG-floor (`floor_ratio_of`/`meets_floor_lc`) | **0** (чистый WCAG на квантованном дисплее) | `solve.rs:970, 1236` |
| target-met | 0 | `solve.rs:1236` |

→ verify роли = **1 форвард** против ~tens на полный солв. Полный verify-проход 20 ролей ≈ 20 форвардов против WARM ~800 → **~40×**, если все кандидаты прошли, до любого fallback.

## Инкременты (каждый — независимо отгружаемый и верифицируемый)

1. **C (бесплатно, первым).** Хроматический LUT/аналитический Stage-1-инверс + табуляция `max_contrast`. Бит-идентично через существующий seed-then-verify (`lut.rs`). Гейт: расширить golden + понизить COLD-пин в `cam16_forwards_per_set_regression_guard` (`solve.rs:1283`). Риск: низкий. Выигрыш: ~2–4× COLD. *(Маргинально — можно пропустить.)*
2. **verify-примитив.** `verify_role_against_bg(served_hex, role, ctx, bg, vc) -> bool` = `finish`-форвард + `meets_floor_lc` + WCAG-floor (переиспользовать `solve.rs:594–619`). Без fast-пути; тест: live-выходы проходят, заведомо-неверные hex'и проваливают. **Ключ безопасности всей главы.**
3. **A-поверхность для anchored-ролей, serve-then-verify.** `Y_hk` breakpoint-таблица для 6 anchored-ролей (Label*, Icon, BorderStrong) — самый чистый 1-D-на-полярность случай. Отдать → verify (#2) → fallback в live при промахе. dJ'/декоративные пока live. Гейт: `chromafast`-стиль свип, (a) каждая отданная роль проходит verify, (b) сет байт-в-байт == live (fallback гарантирует). Риск: средний, **но R1 уже закрыт #2** — неполнота таблицы стоит лишь скорости.
4. **A-поверхность для dJ'-ролей** из бесплатного `J'_bg` (см. invariant-doc, п.3). serve-then-verify через dJ'-аналог (`DJ_BUDGET`, `solve.rs:767`).
5. **(опц.) Доказательство полноты** (Lipschitz-плотность + перечисление floor/polarity-границ), чтобы fallback-rate → ~0. Теперь это *оптимизация производительности с уже стоящим корректностным бэкстопом* — можно откладывать/делать оффлайн. Гейт: длинный исчерпывающий `(Y_hk, Y_wcag)`-свип по реальным дисплейным триплетам.

## Инвариант, который нельзя нарушать

Не отгружать fast-путь, возвращающий иной hex, чем live. Гибрид соблюдает это через **live-fallback**. Если команда когда-нибудь выберет более слабую гарантию «контракт-эквивалентно в пределах ±1 кванта», это должно быть **записанным решением** (рядом с этим файлом) с явным ослаблением golden-гейта в том же коммите — никогда молча.

## Границы / 1000×

«~1000×» достижимо только когда И anchored, И dJ'-поверхности отдаются из таблиц с near-zero fallback (инкременты 3+4+5). Каждый инкремент по отдельности — ~2–40×. C в одиночку не дотянет до 50× (WARM-пол доминируют неустранимые per-role `finish`-форварды — подтверждено forward-count гардом `solve.rs:1283`).
